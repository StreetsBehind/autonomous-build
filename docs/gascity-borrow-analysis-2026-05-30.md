# Borrowing from Gas City — grounded analysis (2026-05-30)

*Supersedes the earlier speculative draft of the same name (`docs/gascity-borrow-analysis-2026-05-30.md`), which was written from gascity's README/package names rather than its source and prescribed the explicitly out-of-scope "become an SDK" direction.*

## TL;DR

The two systems are different kinds of tool and the comparison confirms a clean asymmetry: **we are a policy-rigorous, stateless JS-workflow pipeline** (concern derivation, decidedness/coverage gates, adversarial cross-checks, deterministic topology assertions) while **gascity is a robustness-rich, long-running orchestration substrate** (state-machine reconciler, durable lease/retry queue, crash-loop quarantine, structured gate capture). Our planning/decomposition *policy* is in many ways stronger than theirs; their advantage is entirely in *runtime survival under long autonomous runs*, which is exactly the user's goal. Most of their headline survival mechanisms are daemon-shaped and do not port — but a handful reduce to **durable bd-metadata arithmetic, pure-JS assertions, or cheap shell**, and those are the borrows worth taking. The single highest-leverage one is **salvaging already-merged / gate-passable worker commits on crash-reap instead of discarding them** (`execute-2`): it fixes a real lost-work-and-permanent-strand bug in a 2-day unattended run and needs no daemon. Everything genuinely valuable lands in `workflows/build-batch.js`, `workflows/decompose.js`, `workflows/vision.js`, and the gate hook — never as a resident daemon.

## Side-by-side by stage

### Planning (goal/brief → validated plan)

- **gascity:** No single goal→plan transform. Planning rigor is a *declarative config validation* pass plus a *bounded iterative convergence loop*. `internal/config/validate_semantics.go::ValidateSemantics` returns non-fatal advisory warnings (distinct from `ValidateAgents` hard errors); `internal/config/undecoded.go::suggestKey`/`editDistance` emit "unknown field X (did you mean Y?)" via Levenshtein ≤ 2. `internal/convergence/handler.go` (~L344-359) re-runs a wisp until `GatePass` (`TerminalApproved`) or `wispIteration >= maxIterations` (`TerminalNoConvergence`) — a hard ceiling guaranteeing a terminal verdict; `create.go` validates inputs *before* creating any bead and self-terminates a partial create; `metadata.go::NormalizeVerdict` defaults empty/unknown to `VerdictBlock`; `evaluate.go::ValidateEvaluatePrompt` refuses a config artifact missing its load-bearing substrings.
- **us:** `/vision` (`skills/vision/SKILL.md` + `workflows/vision.js`) is a one-shot gated transform of `vision.md` → `plan.lock.json` + `plan.md` + `tenets.md`. Phase 2 derives concern applicability in pure JS (`deriveApplicability`); Phase 3 fans out one agent per applicable concern over a deep-frozen skeleton; Phase 4 reconciles, runs four pure-JS gates (forward-coverage, reverse-trace, concern-decidedness, musthave-nongoal) + a required/excluded contradiction scan, then builds+validates the lock against `schemas/plan.lock.schema.json` (`validateLock`) before the lone write-agent touches disk.
- **Key divergences:** (1) **Iteration model** — theirs is bounded-iterate-to-terminal-verdict; ours is one-shot-or-block (the `/replan` path is human-driven, not self-bounded). (2) **Severity tiers** — they have advisory-vs-blocking; our plan validation is binary (`incomplete:true` or not). (3) **Validate-before-mutate** — we already share the spirit (Phase 4 fully validates in JS before writing), but we have no resume story for a `/vision` interrupted mid-write.

### Decomposition (plan → beads / work units)

- **gascity:** A deterministic compile-then-atomic-apply pipeline in Go. `internal/formula/compile.go::Compile` (13 stages) produces a `Recipe`; `orderGraphRecipeSteps` runs a Kahn topo-sort and hard-errors "contains a dependency cycle" **at compile time, before any bead exists**; `parser.go::Resolve` rejects circular `extends`. Instantiation is one atomic transaction: `molecule/graph_apply.go::buildRecipeApplyPlan` → `beads/bdstore_graph_apply.go::ApplyGraphPlan` (one hidden `bd create --graph <file> --json`), edges authored against caller-stable `FromKey/ToKey/ParentKey`, then `beads/graph_apply.go::ValidateGraphApplyResult` asserts every planned node Key resolved to a real ID. `molecule.go::Instantiate` classifies transient store faults (`isTransientGraphApplyError`) and retries once.
- **us:** `/decompose` (`workflows/decompose.js` + `.spec.md`) fans out one `pour-feature` agent per feature (`bd mol pour` + reparent + metadata), a separate `wire-cross-deps` agent applies edges one `bd dep add` at a time, then verifies after creation: a Phase 4/5 rework loop to a 95 bar, a Phase 6 adversarial 3-verifier fidelity cross-check, and a Phase 7 dep audit with a **genuinely deterministic pure-JS topology assertion** (`assertTopology`: acyclic / tier-monotonic / reaches-foundation / initial-ready). Pre-pour we validate `feature.vars` against the formula `[vars.*]` contract; cross-dep wiring is verify-after-write (`verifiedPresent`, never trusting `bd dep add` exit code).
- **Key divergences:** (1) **Atomicity** — theirs is all-or-nothing per subgraph; ours is N separate calls (a crash mid-decompose can leave a partial DAG, which our gates detect after the fact). (2) **Cycle-check timing** — they reject cycles at compile time; we discover them post-creation (a cyclic DAG burns a pour+rework before failing). (3) **Edge resolution** — they use symbolic keys resolved atomically; we resolve name→ID in JS (`featureToPourRoot`), the site of the real 3fr.3 "0/28 applied" false-negative.

### Execution (building the work units)

- **gascity:** A controller/reconciler pushes routed work to pools; ephemeral workers self-claim (`bd update --claim`) and verify a two-key match (assignee==session AND routed_to==template) before working (`prompts/pool-worker.md`, `graph-worker.md`). The refinery patrol (`mol-refinery-patrol.toml`) is the single serialized merge processor (rebase, gate, ff-merge+push or reject). Retry is a resumable `gc.retry_state` state machine in `dispatch/ralph.go`; `cmd/gc/crash_tracker.go` quarantines a crash-looping session; the witness (`mol-witness-patrol.toml`) **salvages** orphaned worktrees (commits + pushes, checks if already merged) before returning a bead. Gate output is a structured byte-bounded `GateResult` persisted per attempt (`convergence/gate.go`, `capture.go`).
- **us:** `/build-batch` (`workflows/build-batch.js`) is a single stateless wave-dispatch orchestrator. A candidate-picker greedily selects up to `--workers` beads with pairwise `filesTouched` disjointness; the orchestrator owns serialized `bd --claim` + worktree-create (exit-code-distrust verification). Workers build, run `hooks/post-build-gate.sh`, retry on red up to `escalationBudget.maxFailuresPerTask` (+2 load-bearing), return a marker. Merges are serialized (for-loop, one in flight) with auto-rebase retry on conflict and `git reset --hard HEAD~1` rollback on a red post-merge gate. We have a startup crash-recovery reaper, per-worker `STAGE_TIMEOUT_MIN` race, per-wave checkpointing, a `--budget` stop, and a >50% failure-rate abort.
- **Key divergences:** (1) **Salvage vs discard** — their witness salvages orphaned commits; our reaper `git branch -D`s them ("never passed the post-merge gate"). (2) **Crash-loop handling** — they quarantine a flapping session; we have a failure-*rate* abort but no per-bead restart ceiling across runs. (3) **Gate output** — theirs is structured + byte-bounded per attempt; ours pipes free-form output into a bead note.

### Robustness & recovery (the user's explicit goal)

- **gascity:** Robustness is first-class durable substrate. `internal/convergence/reconcile.go` reconciles forward from per-bead `FieldState` (classifies which step was interrupted, replays the lost transition) keyed by durable idempotency keys (`converge:<bead>:iter:<n>`). `cmd/gc/crash_tracker.go` quarantines after N restarts in a sliding window. `internal/nudgequeue/state.go` is a durable lease/retry/dead-letter queue (`LeaseUntil`, `recoverExpiredInFlightNudges`, dead-letter at `Attempts >= 5`). `reviewquorum/finalize.go` classifies transient (→`VerdictBlocked`, retryable) vs hard (→`VerdictFail`) reviewer failures. `condition.go::RunCondition` retries gate scripts *only* on timeout. `storehealth/storehealth.go` + `reliability/reliability.go` surface store/agent health during a run.
- **us:** Stateless-per-run JS + bd; recovery is mostly "re-run the workflow," but `build-batch.js` has a real recovery surface: a crash-recovery agent resumes processed sets from `STATE_FILE` and reaps stale claims; the serialized merge has auto-rebase + rollback; `post-build-gate.sh::ratchet_eval` treats a missing/unparseable receipt as SKIP (crash never reads as a quality regression); `retro.js` and `decompose.js` run adversarial cross-check quorums; `ESCALATION_RULES.md` bounds retries by `escalationBudget`.
- **Key divergences:** (1) **Recovery model** — they reconcile forward from durable per-item state; we re-run and reap orphans back to open (discarding crashed work). (2) **Idempotency** — they have explicit keys; we dedup by ID-set in `STATE_FILE`, with sub-steps relying on git/bd being naturally idempotent. (3) **Health monitoring** — they watch store/agent health mid-run; we have no equivalent runtime-health signal during a long run.

## Recommended borrows (ranked)

Only `adopt`/`adapt` candidates with confirmed grounding. Sorted by reliability impact (high → low), then fit cost (low → high).

### ☐ Salvage unmerged worker commits on stale-claim reap (`execute-2`) — adapt — effort: M
- **Failure mode:** Over a 2-day unattended run, two real bugs. (1) A builder that crashed *after* committing a gate-passable commit but *before* merge has its commit `git branch -D`'d and rebuilds from scratch — if it crashes the same way each time it can permanently strand a load-bearing bead and its whole downstream subtree. (2) The merge agent's `bd close` is its *last* step, so a crash between `git merge --no-ff` and `bd close` leaves a bead `in_progress` whose work is already on `main`; our reaper re-opens it and a worker wastefully rebuilds it.
- **gascity evidence:** `examples/gastown/packs/gastown/formulas/mol-witness-patrol.toml` (recover-orphaned-beads Cases C/D salvage, Step 3 `git merge-base --is-ancestor` close-already-merged).
- **Landing shape:** `workflows/build-batch.js` crash-recovery reaper (~L191-211). Land the **cheap, high-value half now:** add a local `git merge-base --is-ancestor bead/<id> main` check and `bd close` already-merged orphans instead of re-dispatching. Defer the keep-branch-and-re-gate salvage as a follow-up (our local-only, fresh-branch-per-worker model needs a new merge-loop path, not just flipping the existing `hadUnmergedWork` boolean).
- **Note:** Use *local* refs only (our builds are local-first; no `git push origin HEAD`).

### ☐ Cross-wave crash-loop quarantine for poison beads (`execute-1`) — adapt — effort: S
- **Failure mode:** A deterministically poison bead that hard-crashes the runtime mid-worker (OOM, node restart) is left `in_progress`, reaped to open by the next run, re-picked, and re-crashed — an unbounded reap→repick→recrash loop across orchestrator runs that silently burns the window. (Ordinary worker-*reported* failures already block on first occurrence, so this is narrow to true orchestrator-level crashes.)
- **gascity evidence:** `cmd/gc/crash_tracker.go` (sliding-window restart counter) + the durable form `session_reconcile.go::recordWakeFailure` (persisted `wake_attempts`); `mol-witness-patrol.toml` Step 4 (`metadata.recovered`).
- **Landing shape:** `workflows/build-batch.js` reaper — ~10 lines. On reset of an `in_progress` orphan, read+increment a durable `metadata.reaped` counter; once `>= K` (e.g. 3), set `--status=blocked --notes 'crash-loop: reaped K times without converging'` so it routes to `/escalate`. **Use the durable bd-metadata form, not the in-memory sliding window** (we have no long-running controller to hold it).

### ☐ Transient-vs-permanent retry on the decompose pour path (`decompose-4`) — adapt — effort: S
- **Failure mode:** A single transient jsonl/db lock-contention error on `bd mol pour` (exactly the concurrent-checkout contention MEMORY warns about) drops the whole decompose run to NEEDS-FIX with no retry — the pour path names only *permanent* causes (required-var, off-enum, formula-not-found) and has no transient handling.
- **gascity evidence:** `internal/molecule/graph_apply.go::isTransientGraphApplyError` + `molecule.go::Instantiate` (one timed retry).
- **Landing shape:** `workflows/decompose.js` pour agent (~L254-272) + the wire agent's command-error path. Add a one-shot sleep-and-retry on transient bd error text, mirroring the pattern already in `build-batch.js:103` (bd-doctor-and-retry on the read path). **Classify on real bd error text** (`database is locked`, `lock`, `i/o timeout`, `resource busy`) — *not* gascity's network-fault list. Keep required-var/off-enum/formula-not-found as immediate loud fails.

### ☐ Post-apply completeness invariant: every planned step resolves to a real bead (`decompose-2`) — adapt — effort: S/M
- **Failure mode:** Silent partial pour — `bd mol pour` exits 0 but spawns a subset of declared steps (bd hiccup, jsonl write race), the pour agent returns `status='ok'` with a short `children[]`, the `pourFailed===0` gate passes, and the deficit only *maybe* surfaces later in the probabilistic, token-heavy Phase 6 fidelity verifiers (which check feature-level coverage, not per-formula step presence).
- **gascity evidence:** `internal/beads/graph_apply.go::ValidateGraphApplyResult` (hard-fails on any unresolved node Key), called on every apply path.
- **Landing shape:** `workflows/decompose.js` — have each pour return its formula's **substituted, condition-filtered** expected step-title set alongside `children[]`, then add a pure-JS "expected-but-absent" assertion (mirroring the existing `crossDep verifiedPresent` / `assertTopology` checks) forcing NEEDS-FIX on any deficit. **Must honor formula `condition`/`needs` filtering to avoid false positives, and skip dry-run.**

### ☐ Mid-run store-health watchdog between waves (`robustness-5`) — adapt — effort: S
- **Failure mode:** Silent partial failure / false-drain. A jsonl lock error or the `--db`-at-directory trap silently returns empty results, so over a multi-hour batch a degrading/locked store makes the loop read "no ready work" and either strands the rest of the batch or false-positives a clean drain into `/retro`. Our preflight catches the lock case only at wave 1; the wave loop has no re-probe.
- **gascity evidence:** `internal/storehealth/storehealth.go::Compute` + `internal/doctor/checks_bd_backup_size.go` (cheap periodic health probe + Warning threshold + abort).
- **Landing shape:** `workflows/build-batch.js` — one extra Bash-agent call at the top of each wave (~L332) that re-runs the existing L103 preflight (`bd ready --json`, `bd doctor --fix` once on anomaly), breaking into the existing `[BATCH ABORT]` exit path if still unhealthy. **Adapt the pattern, not the code** — drop gascity's Dolt-specific MB/row+GC metric (meaningless for jsonl), and throttle (every N waves or only on anomaly) so the probe doesn't contend on the jsonl it guards.

### ☐ Idempotency-keyed merge-resume for ready-to-merge beads (`robustness-1`) — adapt — effort: M
- **Failure mode:** Lost work on crash — re-running an expensive builder LLM for a bead that already reached `ready-to-merge` (built@sha) but crashed in the small merge window; the reaper `git branch -D`s the surviving commit and rebuilds from scratch.
- **gascity evidence:** `internal/convergence/handler.go` (`IdempotencyKey`/`FindByIdempotencyKey`) + `reconcile.go` (state-classification, "last write is the idempotency marker").
- **Landing shape:** `workflows/build-batch.js` + recovery-agent prompt. **Skip the full 4-state ledger** (no useful mid-resume points in our model); instead persist `{beadId, builtSha}` per-bead *inside the merge loop* (move checkpoint from once-per-wave to per-bead for ready-to-merge beads), and teach the recovery agent that a surviving `bead/<id>` tip == `builtSha` may resume at the merge agent (which re-runs the gate on main anyway) instead of being branch-D'd and rebuilt. The whole job is **reconciling that carve-out with the reaper's delete-branch rule** — git's `commitSha` is the natural idempotency key (re-merge of the same tree is a no-op). *Overlaps with and should be sequenced after `execute-2`.*

### ☐ Pre-pour cycle check over cross-feature deps (`decompose-3`) — adapt — effort: S
- **Failure mode:** A cyclic `crossFeatureDependencies` set survives to Phase 7, where `assertTopology` forces NEEDS-FIX *after* a full pour+wiring pass already mutated bd — wasted work, fix is a `/vision` re-author. (Capped at med because tier edges are acyclic by construction, so the only cycle source is hand/vision-authored cross-deps, which is small and rare.)
- **gascity evidence:** `internal/formula/compile.go::orderGraphRecipeSteps` (Kahn topo-sort, compile-time cycle reject) + `parser.go::Resolve` (circular `extends`).
- **Landing shape:** Lift the existing `assertTopology` DFS (`workflows/decompose.js:1180-1195`) into a ~15-line pure-JS helper run over the in-memory `ParsedPlan.crossDeps` between Phase 2 and Phase 3 — and ideally also into `vision.js::validateLock` so a cyclic lock never ships. **Scope to `crossFeatureDependencies` only** (tier edges are acyclic by construction; formula-internal deps aren't known pre-pour).

### ☐ Structured, byte-bounded gate-result capture (`execute-3`) — adapt — effort: M
- **Failure mode:** Unbounded gate-output bloat — three `--append-notes` sites paste a verbatim unbounded failing-test/clippy log into a bead note, drowning the actionable failure line and inflating what `/escalate` and `/retro` must read. (Degradation/cost mode, not a correctness mode — the build still blocks correctly.)
- **gascity evidence:** `internal/convergence/gate.go` (`GateResult`), `capture.go` (`MaxOutputBytes=4096`, rune-safe `TruncateOutput`), `dispatch/ralph.go` (`persistCheckResult`, `traceClipString`).
- **Landing shape:** `hooks/post-build-gate.sh` **and** `hooks/post-build-gate.ps1` (dual-port, kept in sync per CLAUDE.md) emit a single ~4KB-bounded JSON tail line (outcome, exitCode, durationMs, rune-safe head+tail of stdout/stderr, truncated bool); convert the 3 raw-paste `--append-notes` sites (`build-batch.js:583/585/590`, `build-next SKILL.md:223`) to write bounded notes + `gate.exit`/`gate.truncated` metadata. **Scope to the gate-output path only** — do not import the full typed-`GateResult` abstraction. *Quality-of-life robustness; queue below the correctness-class borrows.*

### ☐ Bounded iterate-to-terminal-verdict for self-fixable plan blocks (`plan-1`) — adapt — effort: M
- **Failure mode:** Spurious human-gate stops that strand the unattended walk-away window — a flaky concern fan-out reply, an off-enum formula pick, or a reverse-trace orphan halts the whole pipeline at orchestrate Stage 1 and pages the human for something a bounded re-derive could clear. (Med, not high: costs an unattended window, not lost/corrupted work.)
- **gascity evidence:** `internal/convergence/handler.go` (~L344-359 terminal switch) + `create.go` (`max_iterations` must be positive).
- **Landing shape:** `workflows/vision.js` Phase 4 + `vision.spec.md` (T3 lockstep). Adapt the bounded-loop-to-loud-terminal-verdict shape **we already run twice** (`decompose.js` `MAX_REWORK`, `build-batch.js` retry budget) into vision, **scoped narrowly** to mechanically self-fixable blocks (off-enum formula, reverse-trace orphan, evidence-less concern) with a small N ceiling. **Keep product-judgment blocks (missing-product-sections, musthave-nongoal) one-shot human gates.** The re-run must feed prior blocks as targeted context *without* perturbing the frozen-skeleton determinism that vision-eval L3 ratchets on.

### ☐ Stable-key completeness assert on poured features (`decompose-5`) — adapt — effort: S
- **Failure mode:** Silent partial failure / graph drift in cross-dep wiring (same class as the 3fr.3 "0/28 applied" false-negative). `featureToPourRoot` is built with `if (r.pourRoot)`, so a feature that poured ok but lacks a pourRoot — or a name collision — silently drops an edge with no JS assert.
- **gascity evidence:** `internal/beads/graph_apply.go` (`GraphApplyEdge.FromKey/ToKey`, `ValidateGraphApplyResult`).
- **Landing shape:** `workflows/decompose.js` — after building `featureToPourRoot`, add a pure-JS assert that every `status=ok` poured feature resolved to a key (else NEEDS-FIX) and that no two features collided to the same name. **Borrow only this cheap invariant; skip the daemon-only atomic `bd create --graph` substrate** (the stable-key threading itself already shipped as the 3fr.3 fix).

### ☐ Paired-artifact content assertion (lock ↔ tenets.md / plan.md) (`plan-4`) — adapt — effort: S
- **Failure mode:** Silent artifact drift — a `renderTenets`/`renderPlanMd` bug ships a `tenets.md`/`plan.md` whose content diverges from what the lock implies, and the human reviews the prose at the gate while the lock silently disagrees. (Low/latent — no retro shows it firing; the tenets.md side has the real teeth since `renderTenets` reads `frozen.nonGoals`, a *different* source than the lock.)
- **gascity evidence:** `internal/convergence/evaluate.go::ValidateEvaluatePrompt` (refuses an artifact missing its load-bearing substrings).
- **Landing shape:** `workflows/vision.js` — promote the existing fixture spot-check (vision.js:1539) into a per-run pure-JS `validateRenderedArtifacts(lock, planMd, tenetsMd)` (FAILED/write-nothing on miss, slotting into `reconcileAndAssemble` after the renderers). **Scope the assertions to divergent sources** (tenets.md ← `frozen.nonGoals`, plan.md ← lock concern evidence/reason) — *not* the tautological lock→lock loops. *Cheap insurance against a real-but-unobserved render drift.*

### ☐ Two-tier verdict: blocking gates vs non-fatal plan warnings (`plan-2`) — adapt — effort: S/M
- **Failure mode:** No reliability failure mode directly — a plan-quality/auditability gap. Today the only way to make a plan smell visible is to make it blocking (`blockingCompose:true`), which forces a false NEEDS-INPUT the stability harness penalizes, so authors are pressured to leave smells silent. A non-fatal tier removes that bad incentive.
- **gascity evidence:** `internal/config/validate_semantics.go::ValidateSemantics` (advisory warnings, distinct from `ValidateAgents` hard errors).
- **Landing shape:** `schemas/plan.lock.schema.json` (additive optional `warnings[]`, schemaVersion stays 2) + `workflows/vision.js` (conservative pure-JS Phase-4 soft-checks + `renderPlanMd` non-blocking section) + `vision.spec.md`. **Never touches the `incomplete` computation** (so `validateLock`'s incomplete-equals-blockingCompose invariant is untouched). *Schedule as polish — touches none of our actual autonomous-run failure modes and risks gate-noise if warning codes are tuned loosely.*

**Overall assessment:** The prior doc is **low reliability as a basis for decisions** — it reads as written from gascity's README/package names, not its source. Every borrow is a generic SDK-architecture noun (pluggable runtime, pool routing, reconciliation loop, durable messaging, layered packs) with no file/function citation; its thesis prescribes the forbidden "become a general orchestration platform" direction; and it gets gascity's own vocabulary wrong (claim 4's "task/message/session/container" is not the actual bead/wisp/molecule/convoy/session/mail taxonomy). It shows no evidence of reading *our* repo — it recommends as novel several things we already ship (formulas as TOML, `bd mol pour --dry-run` ephemeral mode, reconciliation/stale-claim reaping with checkpointing).
